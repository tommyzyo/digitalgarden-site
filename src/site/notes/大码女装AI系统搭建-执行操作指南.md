## 第 1 周：数据底座搭建

目的：不依赖开发，让人和数据先跑起来。

- [ ] 1.1 建 SKU 表（飞书多维表格）：sku_id、品类码、供应商码、年份-流水、颜色、尺码、进货价、售价、库存。编码规则 `{品类}-{供应商}-{年份}-{流水号}`，例 CS-GX03-26-0012-WH-XL。品类码：CS=衬衫, SK=半裙, TK=裤装, LQ=连衣裙, JK=外套, TQ=套装。
- [ ] 1.2 建身型档案表：customer_id、身高、体重范围、胸围、腰围、臀围、肩宽、在意部位、风格偏好。
- [ ] 1.3 建试穿事件表：customer_id、sku_id、尺码、结果(过/不过)、失败原因、fit_score、日期。
- [ ] 1.4 建内容投放表：content_id、平台、主题、身型标签、SKU、券码、曝光、私信、预约、到店、成交。
- [ ] 1.5 建订单表：order_id、customer_id、sku_id、毛利、来源(平台/内容ID)、日期。
- [ ] 1.6 建供应商评分表：档口、品类、尺码覆盖、补货速度、质量稳定性、退换配合。
- [ ] 1.7 定标签字典：身型标签先 10 个高频（附录 F），SKU 风险标签先 10 个高频（附录 G）。
- [ ] 1.8 预约表单改造：增加来源平台、场景需求、尺码需求、身型简单选项；增加授权勾选框（身型数据 + 营销触达 + 可撤回说明）。
- [ ] 1.9 首周基线测算：记录人工推荐成交率（目标参照线约 25-35%）；记录首月滞销率和断码率。

交付物：
1. 飞书多维表格 x 6 张（SKU、身型、试穿、内容投放、订单、供应商）。
2. SKU 编码规则文档。
3. 标签字典（身型 10 + 风险 10）。
4. 预约表单改版截图。
5. 首次基线数据。

## 第 2 周：手工数据闭环

目的：先证明数据能转成经营动作，不依赖系统。

- [ ] 2.1 店员每天记录试穿结果和失败原因，飞书表每天录入，指定人审核。
- [ ] 2.2 每周导出平台数据，填入内容投放表。
- [ ] 2.3 用 AI 生成第一版周报：用附录 A 模板跑提示词，输出关键指标（新增档案、试穿数、CAC、到店成本、通过率 Top10、失败原因 Top10、补货和淘汰建议）。
- [ ] 2.4 用 AI 生成第一次选款会材料：SKU 试穿通过率排行 + 失败原因分析。
- [ ] 2.5 审批授权文本：身型数据授权协议文案（律师审过）、内容授权协议文案。

交付物：
1. 第 1 周经营周报（含基线数据）。
2. 第一次选款会材料。
3. 授权协议文案定稿。

---

## 第 3-4 周：半自动化 Agent — DB 与 API

目的：把表格搬到数据库，让 Agent 能读数据。

- [ ] 3.1 搭建数据库（PostgreSQL / Supabase / 腾讯云任选），按研究文档第 6 章数据模型建表（customers、skus、try_on_events、orders、content_posts 等），每表含 created_at 和 updated_at。
- [ ] 3.2 飞书表到 DB 同步（n8n/Make 定时导入，或一个 Node.js 导入脚本）。
- [ ] 3.3 接入预约/小程序事件：预约表单提交直接写入 DB。
- [ ] 3.4 接入 POS/库存每日导入：CSV 导入或直接 API，保证订单和库存数据每日更新。
- [ ] 3.5 接入巨量引擎 API：注册开放平台 -> 创建应用 -> 申请 Marketing API 权限。建表 ad_reports、conversion_events，每日拉取投放与素材报表。
- [ ] 3.6 接入腾讯广告 API：保留 qz_gdt/gdt_vid 等点击 ID；预约 -> 到店 -> 成交事件回传。
- [ ] 3.7 接入企业微信客户联系：同步客户标签、跟进人、社群状态。Agent 只输出待跟进任务和话术草稿，不开自动群发。
- [ ] 3.8 打通关键转化 ID 链路：每个内容和投放使用唯一 content_id 或券码；预约 -> 到店 -> 成交链路可还原。

交付物：
1. 数据库部署 + 建表 SQL。
2. 飞书表到 DB 同步脚本。
3. 巨量/腾讯 API 拉取脚本。
4. 企微客户标签同步脚本。
5. 转化链路验证（选一个真实内容测通）。

---

## 第 5-6 周：半自动化 Agent — 工具与看板

目的：让 Agent 输出每日可执行动作，而不是文本。

- [ ] 5.1 建立 Agent 工具函数（建议 Node.js 或 Python 脚本集）：
  - get_funnel_report(date_range, platforms, metrics)
  - get_customer_profile(customer_id)
  - recommend_skus_for_customer(customer_id, scenario, budget_range, max_items, require_in_stock)
  - score_restock(end_date, threshold)
  - score_supplier(date_range)
  - score_pattern_opportunity(body_tag, category, evidence_window_days, minimum_tryon_count)
- [ ] 5.2 写增长 Agent 逻辑：输入内容/投放数据 + 线索分群（高意向未预约/预约未到店/到店未成交/成交未复购），输出跟进任务列表 + 话术草稿 + 裂变活动建议。
- [ ] 5.3 写导购推荐 Agent 逻辑：规则层按尺码+库存+价格硬过滤 -> 身型匹配 -> 场景排序 -> 历史权重。LLM 层生成易懂推荐理由 + 店员话术 + 踩雷提醒。
- [ ] 5.4 写货盘分析 Agent 逻辑：规则层按试穿通过率 + 销量 + 库存周转 + 供应商评分计算。输出补货清单、淘汰清单、替代款需求。
- [ ] 5.5 写投放归因 Agent 逻辑：规则层计算 CPL -> 预约成本 -> 到店成本 -> CAC -> 首单毛利覆盖率。输出各平台 ROI 报告 + 预算调整建议。
- [ ] 5.6 建每日任务看板：简单前端或直接输出到飞书群/企微机器人，每天早晨推送：今日待跟进客户数、今日建议内容主题、今日补货预警。
- [ ] 5.7 建试穿推荐原型：输入顾客 ID + 场景，输出 5-8 个 SKU + 理由。
- [ ] 5.8 建投放归因日报：每日自动推送各平台花费/点击/预约/到店/成交/毛利。

交付物：
1. 所有 Agent 工具函数脚本集。
2. 增长、导购、货盘、投放 Agent 逻辑文件。
3. 每日看板（机器人消息或简单 Web 页面）。
4. 推荐原型可演示（输入 -> 输出）。
5. 归因日报可查看。

## 第 7-8 周：可融资 Demo — 驾驶舱与验真

目的：把零散 Agent 和数据拼成投资人能看懂的 Demo。

- [ ] 7.1 建经营驾驶舱（一页看板）：Metabase / Superset / 飞书仪表盘。板块包括获客漏斗、试穿漏斗、成交漏斗、货盘健康、版型机会、Agent 采纳率。数据按周更新。
- [ ] 7.2 做 Agent 采纳追踪：每个 Agent 推荐 -> 人工选采纳/不采纳 -> 记录结果。可输出"增长Agent采纳率 82%，采纳后预约率提高 15%"。
- [ ] 7.3 版型机会评分表登场：每周自动计算 Pattern Opportunity Score（公式见源文档第 10 章）。输出可视化排名。
- [ ] 7.4 做 B2B 选品原型：输入城市+客群特征+价格带，输出拿货组合清单 + 尺码配比 + 话术。手工验证 1-2 个小店老板。
- [ ] 7.5 做合规展示：系统可展示授权记录、撤回入口、脱敏字段示例。
- [ ] 7.6 写一页投资人技术页：标题 AI-driven Fit Commerce Stack；三个壁垒（真实身型数据、SKU 适配数据、经营 Agent）；三个验证指标（推荐成交率 vs 基线、CAC vs 首单毛利、微改版通过率）。

交付物：
1. 经营驾驶舱（页面/截图）。
2. Agent 采纳率统计。
3. 版型机会评分表（周更）。
4. B2B 选品原型演示。
5. 合规展示截图。
6. 投资人技术页（PDF）。

---

## 第 9-12 周：可融资 Demo — 完整闭环

目的：一次完整轮转——数据进来 -> Agent 出建议 -> 人工做 -> 结果记回去。

- [ ] 9.1 内容主题闭环验证：选一条内容，全链路追溯（曝光 -> 私信 -> 预约 -> 到店 -> 成交 -> 毛利）。
- [ ] 9.2 SKU 闭环验证：选一个 SKU，全链路追溯（实量 -> 试穿通过率 -> 失败原因 -> 补货建议 -> 是否微改版 -> 结果）。
- [ ] 9.3 身型标签闭环验证：选一个身型标签（如"胸大"），展示适合的 SKU、缺口品类、相关内容主题、版型机会。
- [ ] 9.4 Agent 日常运营固化：增长 Agent 每天自动出跟进任务，导购 Agent 生成推荐（店员测试），货盘 Agent 每周四出补货/淘汰建议。
- [ ] 9.5 供应商评分投入使用：基于补货速度 + 质量 + 退换 + 爆款贡献，每周评一次并推送。
- [ ] 9.6 微改版验证表：记录 3-5 个版型机会的打样 -> 试穿 -> 结果。
- [ ] 9.7 180 天指标快照：记录当前到 12 周的里程碑数据（身型档案数、试穿事件数、CAC、推荐成交率、复购率、转介绍占比）。
- [ ] 9.8 交付物归档：所有代码/脚本/使用说明放一个 git repo；写操作 SOP 文档给店员。

交付物：
1. 完整闭环 Demo（可录视频）。
2. 供应商评分周报（正式投入使用）。
3. 微改版验证表。
4. Git repo + 操作 SOP。
5. 180 天指标快照（第 12 周）。

## 第 13-26 周：指标冲刺

目的：不增新功能，专注拉平 180 天里程碑。

- [ ] 身型档案达到 800-1500
- [ ] 试穿事件达到 3000+
- [ ] 结构化 SKU 达到 300-500
- [ ] 导购 AI 推荐成交率高于人工基线 5-10 个百分点
- [ ] 滞销率比基线降 15%+
- [ ] 断码率比基线降 20%+
- [ ] 版型机会 10-20 个，微改版验证 3-5 个
- [ ] 复购率 30/60/90 天可输出
- [ ] B2B 拿货包至少 2 个实际客户验证
- [ ] 每周出具经营技术周报（投资人可看）

## 附录 J：异常预警阈值

| 指标 | 预警阈值 | 动作 |
| --- | --- | --- |
| 内容到私信率 | 低于 3% | 换标题/封面 |
| 私信到预约率 | 低于 10% | 改话术/券码力度 |
| 预约到店率 | 低于 50% | 增加到店提醒/送券 |
| 试穿成交率 | 低于 20% | 检查推荐匹配度 |
| 试穿失败率超过 50% 的 SKU | 标记为应淘汰 | 报选款会 |
| CAC > 首单毛利 | 停止该渠道/素材投入 | 报运营负责人 |
| 到店成本 > 客单价 30% | 调整投放预算分配 | 报运营负责人 |
| 供应商连续 2 周补货延迟 | 找替代 | 报供应链负责人 |
